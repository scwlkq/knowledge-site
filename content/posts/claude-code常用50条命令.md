# Claude Code 50+ 斜杠命令完全指南

> 掌握这些命令，让你的 AI 编程效率翻倍

## 前言

Claude Code 内置了 50 多个斜杠命令，覆盖会话管理、上下文控制、并行协作等方方面面。虽然你可以直接用中文跟 Claude 说需求，它会自动执行对应命令，但了解这些命令的作用和使用场景，能让你知道 Claude Code 的能力边界在哪里，打开使用思路，提高效率。

本文将按分类详细介绍这些命令的用法、使用场景和最佳实践，并加入我的实战经验和理解。

**为什么要学这些命令？**

作为一个深度使用 Claude Code 的开发者，我发现很多人把 AI 编程工具当成"聊天机器人"来用，这其实浪费了它 80% 的能力。Claude Code 的命令系统就像 Git 的命令一样，是专门为编程场景设计的工作流工具。掌握这些命令，你会发现：

1. **效率提升 3-5 倍** - 不用反复解释需求，一个命令搞定
2. **成本降低 50%** - 通过 `/compact` 等命令优化上下文使用
3. **代码质量更高** - `/review` 和 `/simplify` 自动发现问题
4. **工作流更清晰** - `/plan` + `/goal` 让大型任务井井有条

## 一、会话管理命令

### 1. `/clear` - 清空对话

**分类标签**：#会话管理 #上下文重置

**作用**：清空当前对话，开始全新会话。

**使用场景**：完成一个任务，要切换到下一个任务时。

**注意**：清空的对话不会丢失，可以通过 `/resume` 找回。如果只是觉得上下文太长，应该用 `/compact` 而不是 `/clear`。

```
/clear
```

---

### 2. `/compact` - 压缩对话 ⭐

**分类标签**：#会话管理 #上下文管理 #成本控制

**作用**：压缩当前对话，释放上下文空间。

**使用场景**：上下文用量超过 80% 时，或完成阶段性任务后。

**原理**：把之前的对话浓缩成摘要，腾出空间继续工作，同时保留关键信息。压缩后每次请求发送的 token 也会减少，相当于间接省钱。

**💡 我的理解**：
这是我使用频率最高的命令之一。很多人不知道，Claude 的上下文窗口虽然有 200K tokens，但当使用率超过 90% 时，AI 的"注意力"会明显下降，就像人在信息过载时一样，容易遗漏细节。

我的习惯是：
- 每完成一个功能模块就 `/compact` 一次
- 在开始新的大任务前先压缩
- 如果发现 AI 开始"健忘"（忘记之前说过的事），立刻压缩

**最佳实践**：
- 不要等到自动压缩（95%）才处理
- 在每完成一个阶段性任务后主动执行
- 可以加指令告诉它重点保留什么
- 压缩前先用 `/context` 看看哪些内容占用最多

```
/compact
/compact 重点保留 API 接口的设计决策和参数定义
/compact 保留所有关于数据库 schema 的讨论，其他可以简化
```

**省钱技巧**：定期压缩可以让每次请求的 token 数减少 30-50%，长期下来能省不少钱。

---

### 3. `/resume` - 恢复会话

**分类标签**：#会话管理 #历史会话

**作用**：恢复之前的会话。

**使用场景**：想继续之前的任务，或查看历史对话。

**技巧**：
- 直接输入会弹出会话选择器
- 按 `Ctrl+A` 切换到全部项目的历史会话
- 可以直接指定会话名字或 ID

```
/resume
/resume 昨天的重构任务
```

---

### 4. `/branch` 或 `/fork` - 分支会话

**分类标签**：#会话管理 #分支探索 #方案对比

**作用**：从当前对话分支出新会话（两个命令完全相同）。

**使用场景**：想基于当前讨论探索不同方向，但不想影响当前对话进度。

**典型场景**：讨论架构方案时，想试试另一种实现思路，又怕改着改着原来的思路找不回来。

**💡 我的理解**：
这个命令就像 Git 的分支功能，但用在对话上。我经常在这些场景下使用：

1. **技术选型对比** - 在主会话讨论方案 A，分支出去试试方案 B，最后对比哪个更好
2. **风险探索** - 想尝试一个激进的重构方案，但不确定是否可行，先分支出去试试
3. **学习实验** - 正在做项目，突然想学习某个新技术，分支出去玩玩，不影响主线

**实战技巧**：
- 分支名字要有意义，方便后续 `/resume` 时识别
- 可以同时维护多个分支，探索不同方案
- 分支之间是独立的，互不影响

```
/branch 试试用 Redis 替代内存缓存的方案
/branch 探索 GraphQL 替代 REST API 的可能性
/branch 学习一下 Rust 的异步编程
```

### 5. `/rewind` - 回退对话

**分类标签**：#会话管理 #回退恢复

**作用**：把对话和代码回退到之前的某个节点。

**使用场景**：Claude 改错了代码，想回到之前的状态。

**原理**：Claude 会自动保存检查点（每次你按回车前的状态），执行后会弹出节点选择器。

**理解**：类似 Word 的撤销，或 Git 的版本回滚，同时回退对话记录和文件改动。

```
/rewind
```

### 6. `/recap` - 会话摘要

**分类标签**：#会话管理 #摘要回顾

**作用**：生成当前会话的一句话摘要。

**使用场景**：离开一段时间后回来，快速回忆做了什么。

**注意**：Claude Code 在你长时间离开后回来也会自动显示 recap，这个命令是让你随时主动触发。

```
/recap
```

### 7. `/btw` - 临时提问

**分类标签**：#会话管理 #临时提问 #上下文隔离

**作用**：快速插入问题，不会污染当前对话上下文。

**使用场景**：正在做任务时，突然想问一个不相关的小问题。

**原理**：相当于开了一个临时小窗，问完就关，不会占用上下文空间或干扰后续任务。

**💡 我的理解**：
这是一个被严重低估的命令！很多人不知道，每次你问 AI 问题，这个问题和答案都会占用上下文空间。如果你在做一个复杂的重构任务，中途问了 10 个不相关的小问题（比如"Python 的装饰器怎么写"），这些内容会一直留在上下文里，干扰 AI 对主任务的理解。

**我的使用场景**：
- 快速查询语法：`/btw TypeScript 的泛型约束怎么写？`
- 概念确认：`/btw Redis 的 ZADD 命令是干什么的？`
- 快速计算：`/btw 帮我算一下 1024 * 768 * 4 字节是多少 MB`
- 翻译查询：`/btw "idempotent" 这个词怎么翻译？`

**对比**：
- 不用 `/btw`：问题和答案会永久留在上下文，占用空间
- 使用 `/btw`：问完就清除，不留痕迹

```
/btw Python 的 dataclass 怎么设默认值？
/btw JWT token 的过期时间一般设置多久？
/btw 帮我解释一下什么是 CORS
```

### 8. `/copy` - 复制回复

**分类标签**：#会话管理 #内容复制

**作用**：把 Claude 的回复内容复制到剪贴板。

**用法**：
- 直接输入复制最新回复
- 如果回复里有代码块，会弹出选择器
- 加数字复制第 N 条回复

```
/copy          # 复制最新回复
/copy 2        # 复制倒数第二条
```

### 9. `/export` - 导出对话

**分类标签**：#会话管理 #对话导出

**作用**：把当前整个对话导出为纯文本文件。

**使用场景**：保存对话记录或分享给同事。

**用法**：
- 直接输入会让你选择复制到剪贴板还是保存到文件
- 加文件名直接保存

```
/export
/export 开发记录.txt
```

### 10. `/exit` - 退出会话

**分类标签**：#会话管理 #退出

**作用**：退出 Claude Code 会话（连按两次 `Ctrl+C` 也能退出）。

**注意**：如果有子代理或后台任务在运行，退出时会询问是否一并关闭。

```
/exit
```

## 二、信息与诊断命令

### 11. `/usage` - 查看用量

**分类标签**：#信息诊断 #用量统计

**作用**：查看当前会话的 token 用量和费用。

**使用场景**：想知道花了多少钱，离计划用量上限还有多远。

**注意**：如果通过 API Key 接入第三方模型（如 DeepSeek），显示的费用可能不准确，以模型供应商后台账单为准。

```
/usage
```

---

### 12. `/context` - 上下文可视化

**分类标签**：#信息诊断 #上下文可视化

**作用**：可视化当前上下文窗口的使用情况。

**显示内容**：彩色网格显示上下文空间被什么占满了（对话记录、加载的文件、系统提示词等），快满了会给出优化建议。

**用法**：
- 直接输入查看概览
- 加 `all` 展开详细信息

```
/context
/context all
```

**最佳实践**：配合 `/compact` 使用，先看 `/context` 确认上下文压力，再决定要不要压缩。

### 13. `/diff` - 查看代码变更

**分类标签**：#信息诊断 #代码变更

**作用**：打开交互式 diff 查看器，看 Claude 改了什么。

**功能**：
- 展示所有未提交的代码变更
- 左右键切换查看整体 git diff 和每一轮的单独改动
- 上下键浏览不同文件

**最佳实践**：在 Claude 改了很多文件后，提交代码前用 `/diff` 过一遍。不过终端看 diff 体验一般，建议配合 VS Code 的 Git 插件。

```
/diff
```

**作用**：打开设置界面的状态标签页，显示版本号、使用的模型、账户信息和连接状态。

**优点**：不需要等 Claude 当前回复结束就能用，随时可以查看状态。

```
/status
```

**分类标签**：#信息诊断 #帮助

**作用**：显示所有可用命令的列表和简短说明。

**使用场景**：记不住命令时随时查询。

/insights

## 三、模型和模式控制命令

### 17. `/plan` - 规划模式

**分类标签**：#模型模式 #规划模式

**作用**：进入规划模式，让 AI 先制定执行计划，确认方案后再动手实现。

**使用场景**：开发全栈项目或做大改动之前，不建议直接让 AI 写代码。

**流程**：
1. AI 可能会通过交互式提问确认更多信息
2. 输出详细的实施方案
3. 等你确认后再开始执行

```
/plan 重构整个项目，增加用户系统和 JWT 认证
```

---

### 18. `/goal` - 目标模式 ⭐⭐⭐

**分类标签**：#模型模式 #目标模式 #自动执行

**作用**：设定完成条件，让 Claude Code 自动持续工作，不需要每一轮都按回车确认。

**使用场景**：有明确终止条件、手动盯着比较费时的任务。特别适合睡觉前丢一个任务，第二天早上起来验收成果。

**原理**：设置 goal 后，每轮结束会有轻量评估模型检查条件是否满足，没满足就自动开始下一轮。

**💡 我的深度理解**：
这是 Claude Code 最强大的命令，没有之一。它彻底改变了我的工作方式。

**为什么这么强大？**
传统的 AI 编程是"对话式"的：你说一句，AI 做一步，然后停下来等你确认。这种方式的问题是：
1. 你必须一直盯着屏幕
2. 简单重复的任务也要不断按回车
3. 晚上睡觉时 AI 就闲着了

`/goal` 打破了这个限制。你只需要告诉 AI "最终要达到什么状态"，它就会自己一直干到完成。

**我的真实使用案例**：
1. **睡前任务** - 晚上 11 点：`/goal 把整个项目从 Python 2 迁移到 Python 3，直到所有测试通过，50 轮后停止`。第二天早上起来，迁移已经完成。
2. **批量重构** - `/goal 把所有 class 组件改成 function 组件，直到 src/ 目录下没有 class 组件`
3. **测试修复** - `/goal 修复所有失败的测试，直到 pytest 退出码为 0`

**关键点 - 如何编写完成条件**：
- ✅ 条件必须是 Claude 执行过程中能自证的
- ✅ 比如"跑某个命令的结果是什么"
- ❌ 不能是主观的（"代码质量要好"）

**好的条件示例**：
```
运行 npm test 退出码为 0，且 git status 没有未提交的更改
src/ 目录下不再有任何 TODO 注释
grep -r "console.log" src/ 没有任何输出
所有 .py 文件都通过 black 格式化检查
```

**坏的条件示例**：
```
❌ 代码质量要好（太主观）
❌ 性能要快（无法量化）
❌ 用户体验要好（AI 无法验证）
```

**最佳实践**：
- 在条件后加熔断限制，避免无限循环
- 不是什么任务都适合用它，token 消耗较大
- 条件要具体、可验证、可量化
- 建议先用 `/plan` 规划，再用 `/goal` 执行

**成本控制**：
`/goal` 会持续消耗 token，所以要设置合理的熔断条件。我的经验是：
- 简单任务：10-15 轮
- 中等任务：20-30 轮
- 复杂任务：40-50 轮
- 超过 50 轮的任务，建议拆分

```
/goal 修复整个项目的代码，直到全部测试通过且没有报错
/goal 把所有 API 调用迁移到 v2 格式，直到测试通过，如果 20 轮还没搞定就停下来
/goal 清理所有 TODO 注释，直到 grep -r "TODO" src/ 没有输出，15 轮后停止
```

**查看进度和停止**：
```
/goal          # 查看当前任务进度
/goal clear    # 提前停止
```

**适用场景**：
- 模块迁移：把旧 API 调用全部迁移到新版本
- 批量重构：拆分大文件，直到每个文件都在指定行数以内
- Bug 修复：修复某个测试用例，直到它通过
- 清理积压：处理标了特定 label 的 issue
- 代码规范：统一代码风格，直到通过 linter 检查

**⚠️ 注意事项**：
1. 不要在生产环境使用 `/goal`，风险太大
2. 第一次使用建议设置较小的轮数限制（10-15 轮）
3. 复杂任务建议先在分支上试验
4. 定期检查进度，避免跑偏

## 五、代码审查命令

### 26. `/review` - PR 审查

**分类标签**：#代码审查 #PR审查

**作用**：让 Claude 审查当前分支对应的 Pull Request。

**流程**：
1. 输入命令后，AI 自动检测当前分支关联的 PR
2. 启动多个子代理并行检查代码
3. 重点找 Bug 和逻辑错误

**使用场景**：在让同事 review 之前或合并之前，自己先用 AI 过一遍，提前发现明显问题。

```
/review        # 审查当前分支的 PR
/review 42     # 审查指定 PR 号
```

**进阶**：`/ultrareview` 命令提供更深度的云端多代理审查，但会额外消耗 usage credits。

---

### 27. `/simplify` - 代码优化

**分类标签**：#代码审查 #代码优化

**作用**：审查最近修改的文件，从三个维度找问题并直接修复。

**三个维度**：
1. 代码复用：有没有重复逻辑可以抽取
2. 代码质量：有没有 hack 写法和冗余状态
3. 执行效率：有没有不必要的计算、可以并发的地方

**与 `/review` 的区别**：
- `/review` 只给意见
- `/simplify` 直接动手改代码

**原理**：并行启动三个审查代理，分别从上面三个角度检查，然后汇总问题并应用修复。

```
/simplify
/simplify 重点关注内存效率
```

## 六、子代理和并行命令

### 28. `/agents` - 查看代理

**分类标签**：#子代理并行 #代理配置

**作用**：查看和配置子代理。

**说明**：Claude Code 可以把子任务委派给子代理并行处理，这个命令用来查看和配置这些子代理。

**使用频率**：不需要手动配置，Claude Code 在需要时会自动启用。这个命令主要是让你看看当前有哪些代理在跑、状态如何。

```
/agents
```

### 29. `/tasks` - 任务管理

**分类标签**：#子代理并行 #任务管理

**作用**：查看和管理当前会话里正在运行的后台任务。

**功能**：
- 查看所有子代理各自在干什么、进度如何
- 选中某个任务按回车，进入查看完整执行过程和输出详情

```
/tasks
```

---

### 30. `/background` - 后台运行

**分类标签**：#子代理并行 #后台运行

**作用**：把当前整个会话放到后台去跑，释放你的终端。

**使用场景**：让 Claude 跑一个比较耗时的任务（重构一个大模块），不想一直开着终端等它。

**流程**：
1. 执行后，当前会话从终端分离，在后台继续运行
2. 终端空闲，可以去做别的事情
3. 想看进度时，用 `claude agents` 命令监控所有后台会话
4. 选中某个后台会话，按右方向键进入对话界面

```
/background
/background 编写单元测试并运行，执行失败则自动修复
```

---

### 31. `/loop` - 循环任务

**分类标签**：#子代理并行 #循环任务

**作用**：设置时间间隔，让 Claude 每隔一段时间执行一次某个操作（定时任务）。

**使用场景**：需要持续监控的场景，比如等部署完成、等 CI 跑完、定期检查日志有没有异常。

**用法**：
- 指定间隔：`/loop 5m 检查部署状态`
- 不指定间隔：Claude 根据任务性质自己决定执行节奏
- 不指定提示词：读取 `.claude/loop.md` 文件内容执行

```
/loop 5m 检查项目前后端的部署状态
```

**注意**：不太建议用 `/loop` 跑长时间定时任务，如果忘了关，会一直在后台循环执行，不知不觉消耗资源。

## 十、常见问题和解决方案

### Q1: 为什么 `/goal` 一直循环，停不下来？

**原因**：完成条件写得不够具体，或者无法验证。

**解决方案**：
- 检查条件是否可量化
- 添加熔断限制（如"20 轮后停止"）
- 用 `/goal clear` 强制停止

---

### Q2: `/compact` 后，AI 忘记了重要信息怎么办？

**原因**：压缩时没有指定保留重点。

**解决方案**：
- 使用 `/compact 重点保留 XXX`
- 重要信息写到项目的 CLAUDE.md 文件中
- 或者用 `/export` 导出后重新加载

---

### Q3: `/review` 审查不够深入怎么办？

**原因**：默认的 `/review` 是快速审查。

**解决方案**：
- 使用 `/ultrareview` 进行深度审查（需要额外费用）
- 或者在 `/review` 后加具体要求：`/review 重点检查安全问题`

---

### Q4: 命令太多记不住怎么办？

**解决方案**：
- 不需要记，直接用中文说需求，AI 会自动执行对应命令
- 常用的就那 10 个，用几次就记住了
- 随时 `/help` 查询
